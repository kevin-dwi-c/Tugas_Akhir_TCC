1. Executive Summary & Problem Statement 
 
1.1 Latar Belakang & Urgensi Klinis 
"Setiap 2 detik, seseorang di Indonesia membutuhkan transfusi darah. Namun proses 
mendapatkan darah yang tepat di waktu yang tepat masih menjadi tantangan kritis di 
lapangan." 
 
Ketersediaan darah yang cepat dan tepat golongan merupakan faktor penentu keselamatan jiwa dalam 
kasus bedah darurat, trauma, komplikasi persalinan, dan penyakit kronis seperti thalassemia. Saat ini, 
sistem distribusi darah di Indonesia masih menghadapi beberapa permasalahan kritis berikut: 
 
• Ketergantungan pada metode manual: Permintaan darah dari rumah sakit ke PMI masih 
dilakukan via telepon atau WhatsApp, rentan terhadap delay komunikasi dan human error dalam 
pencatatan. 
• Minimnya transparansi stok: Pendonor potensial tidak memiliki akses real-time terhadap kondisi 
kekritisan stok darah di PMI terdekat, sehingga motivasi donor sukarela tidak terpicu secara 
optimal. 
• Proses check-in donor yang lambat: Saat kondisi darurat dan antrian donor membludak, proses 
verifikasi identitas dan kelaikan medis secara manual menghabiskan waktu kritis yang harusnya 
digunakan untuk prosedur pengambilan darah. 
• Fragmentasi data donor: Data riwayat donasi tidak terpusat, sehingga validasi jeda donor 
minimum (2 bulan) tidak bisa diverifikasi secara otomatis dan akurat. 
• Jangkauan notifikasi terbatas: Tidak ada mekanisme untuk menjangkau pendonor dalam radius 
geografis tertentu dengan golongan darah spesifik secara masif dan cepat. 
 
1.2 Visi Produk 
Membangun ekosistem digital yang menghubungkan PMI, rumah sakit, dan pendonor secara real-time 
— sehingga respons terhadap kondisi gawat darurat darah dapat berlangsung dalam hitungan menit, 
bukan jam. 
 
1.3 Solusi yang Ditawarkan 
Aplikasi Bank Darah adalah sistem dua sisi (two-sided platform) yang terdiri dari: 
 
• Web App Admin PMI: Dashboard terpusat untuk manajemen stok, penerimaan permintaan 
darurat dari RS, penerbitan broadcast notifikasi ke pendonor terkualifikasi, dan sistem check-in 
donor berbasis QR Code. 
• Mobile App Pendonor (Flutter): Aplikasi untuk menerima notifikasi darurat golongan darah 
spesifik, merespons panggilan, memantau kelaikan donor personal, dan berbagi informasi urgensi 
ke jaringan sosial. 
 
Keunggulan utama sistem ini adalah arsitektur hybrid SQL + NoSQL yang memisahkan data 
sensitif transaksional (SQL) dengan data real-time dinamis (NoSQL), memungkinkan broadcast

1.4 Scope & Batasan Produk 
Dalam scope (v1.0): 
• Manajemen stok darah per golongan dan produk darah 
• Sistem emergency broadcast dengan filter golongan darah + radius 10 KM 
• Kartu donor digital berbasis QR Code 
• Countdown masa siap donor 
• Progress bar stok makro untuk pendonor 
• Fitur share notifikasi ke WhatsApp 
 
Di luar scope (v1.0): 
• Sistem manajemen inventaris kantong darah individual (tracking per kantong) 
• Integrasi langsung dengan sistem HIS (Hospital Information System) 
• Pembayaran atau insentif digital untuk pendonor 
• Telemedis atau konsultasi kesehatan

4. Workflow & System Architecture 
 
4.1 Alur Penanganan Kondisi Gawat Darurat (End-to-End) 
 
Diagram alur berikut menggambarkan proses lengkap dari telepon RS hingga pendonor 
melakukan check-in QR di PMI. 
 
FASE 1 – Inisiasi Permintaan (0–2 menit) 
• Rumah Sakit menelepon PMI dan melaporkan kebutuhan darah mendesak (golongan, jumlah, 
urgensi). 
• Admin PMI membuka Web App → Menu "Permintaan Darurat" → Isi formulir permintaan (FR-
A02). 
• Sistem membuat record baru di tabel SQL blood_requests dengan status PENDING. 
 
FASE 2 – Filter & Broadcast (2–3 menit) 
• Backend men-query SQL: SELECT * FROM users WHERE blood_type = [target] AND is_eligible 
= TRUE AND ST_Distance(location, pmi_location) <= 10000. 
• Sistem mengambil daftar donor_ids yang eligible. 
• Admin mereview jumlah hasil filter → Klik "Kirim Broadcast" → Sistem membuat dokumen baru di 
koleksi NoSQL emergency_broadcasts dengan status ACTIVE dan array eligible_donor_ids. 
• Cloud Function / Backend worker mem-trigger Firebase Cloud Messaging (FCM) ke semua 
device token yang terhubung dengan donor_ids tersebut. 
• Notifikasi dikirim berdasarkan tingkat urgensi: CRITICAL = high-priority FCM dengan sound 
alarm, URGENT/NORMAL = default priority. 
 
FASE 3 – Respons Pendonor (2–30 menit) 
• Pendonor menerima notifikasi → Buka app → Halaman detail darurat. 
• Pendonor tap "Saya Siap Donor" → App menulis dokumen ke koleksi NoSQL live_responses: { 
broadcast_id, donor_id, status: ACCEPTED, timestamp, current_location (opsional) }. 
• Live dashboard Admin diperbarui real-time via Firestore listener (onSnapshot) tanpa perlu refresh. 
• Pendonor yang menolak → Status DECLINED dicatat di live_responses untuk analitik. 
 
FASE 4 – Check-in di PMI (Momen kritis) 
• Pendonor tiba di PMI → Buka halaman "Kartu Donor" di app → QR Code ditampilkan. 
• Admin mengaktifkan QR Scanner di Web App → Arahkan kamera ke QR. 
• Frontend mendekripsi QR → Ekstrak UUID pendonor → Kirim GET /api/donors/:uuid ke Backend. 
• Backend mengambil data lengkap dari SQL: profil, riwayat donasi, status kelaikan → Kirim ke 
Frontend. 
• Admin Web melihat profil donor muncul otomatis → Lakukan verifikasi medis quick-check (TD, 
HB). 
• Admin klik "Konfirmasi Check-in" → Backend INSERT ke SQL tabel donation_history dengan 
status CHECKED_IN.

5. Technical & Database Specifications 
 
5.1 Skema Tabel SQL (RDBMS) 
Rekomendasi: PostgreSQL 15+ dengan ekstensi PostGIS untuk geospatial query. 
 
Tabel: users (Data Pendonor) 
CREATE TABLE users ( 
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(), 
  nik             VARCHAR(16) UNIQUE NOT NULL, 
  full_name       VARCHAR(100) NOT NULL, 
  email           VARCHAR(100) UNIQUE, 
  phone           VARCHAR(15) NOT NULL, 
  blood_type      ENUM('A+','A-','B+','B-','O+','O-','AB+','AB-') NOT NULL, 
  birth_date      DATE NOT NULL, 
  gender          ENUM('M','F') NOT NULL, 
  address         TEXT, 
  latitude        DECIMAL(10,8) NOT NULL,   -- Koordinat domisili statis 
  longitude       DECIMAL(11,8) NOT NULL, 
  location        GEOGRAPHY(POINT,4326),    -- PostGIS column untuk geospatial query 
  photo_url       VARCHAR(500), 
  device_token    VARCHAR(500),             -- FCM token untuk push notif 
  last_donation   DATE, 
  next_eligible   DATE GENERATED ALWAYS AS (last_donation + INTERVAL '60 days') STORED, 
  is_eligible     BOOLEAN DEFAULT TRUE, 
  is_active       BOOLEAN DEFAULT TRUE, 
  created_at      TIMESTAMPTZ DEFAULT NOW(), 
  updated_at      TIMESTAMPTZ DEFAULT NOW() 
); 
 
-- Index untuk performa geospatial query 
CREATE INDEX idx_users_location ON users USING GIST(location); 
CREATE INDEX idx_users_blood_type ON users(blood_type); 
CREATE INDEX idx_users_eligible ON users(is_eligible, next_eligible); 
 
Tabel: hospitals (Data Rumah Sakit) 
CREATE TABLE hospitals ( 
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(), 
  name            VARCHAR(200) NOT NULL, 
  address         TEXT, 
  latitude        DECIMAL(10,8), 
  longitude       DECIMAL(11,8), 
  pic_name        VARCHAR(100), 
  pic_phone       VARCHAR(15), 
  email           VARCHAR(100), 
  is_active       BOOLEAN DEFAULT TRUE, 
  created_at      TIMESTAMPTZ DEFAULT NOW() 
); 
 
Tabel: blood_requests (Permintaan Darurat) 
CREATE TABLE blood_requests ( 
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(), 
  hospital_id     UUID REFERENCES hospitals(id), 
  requested_by    VARCHAR(100),             -- Nama PIC RS yang menelepon 
  blood_type      ENUM('A+','A-','B+','B-','O+','O-','AB+','AB-') NOT NULL, 
  product_type    ENUM('WB','PRC','FFP','THROMBOCYTE') DEFAULT 'PRC',

Data Medis at Rest

Autentikasi API

Token) dengan expiry 24 jam untuk mobile app dan 8 
jam untuk web admin. Refresh token dengan rotasi 
otomatis. 
Security 
Rate Limiting 
Endpoint broadcast dibatasi 10 request/menit per admin 
ID untuk mencegah spam broadcast. Endpoint QR scan 
dibatasi 30 scan/menit per admin session.

Latency

API Response Time 
95th percentile response time API ≤ 300ms untuk

(POST/PUT) di bawah beban normal. 
Performance 
QR Scan Response 
Setelah QR di-scan, profil donor harus muncul di layar

render). 
Performance 
Offline QR Code

tanpa koneksi internet (data UUID disimpan lokal setelah

Reliability 
Uptime Target 
Sistem web backend dan database menargetkan uptime 
99.5% (≈ 3.65 jam downtime/bulan). Notifikasi darurat

Reliability 
Broadcast Spike 
Handling 
Sistem harus mampu menangani broadcast ke 500+ 
pendonor secara simultan tanpa degradasi performa. 
Implementasi FCM message batching dan queue worker 
(BullMQ/Redis). 
Reliability 
Database Failover 
SQL database menggunakan konfigurasi Primary-
Replica dengan automatic failover ≤ 30 detik. NoSQL 
Firestore menyediakan built-in HA. 
Scalability 
Horizontal Scaling 
Backend dirancang stateless (12-factor app) sehingga 
dapat di-scale horizontal melalui container orchestration 
(Docker + Kubernetes / Cloud Run).

8.2 Glossary 
• PMI: Palang Merah Indonesia – organisasi yang mengelola bank darah di Indonesia. 
• PRC (Packed Red Cells): Komponen darah merah yang paling umum ditransfusikan. 
• WB (Whole Blood): Darah lengkap sebelum diproses. 
• FFP (Fresh Frozen Plasma): Komponen plasma darah. 
• FCM (Firebase Cloud Messaging): Layanan push notification dari Google. 
• UUID: Universally Unique Identifier – ID unik yang digunakan sebagai identifier pendonor dalam 
QR Code. 
• Geospatial Query: Query database yang melibatkan kalkulasi koordinat geografis dan jarak. 
• NPS (Net Promoter Score): Metrik kepuasan pengguna skala -100 hingga +100.
